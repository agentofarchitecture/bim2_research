## Pattern update

The source graph suggests three BIM 2.0 layers that matter for AoA:

1. **Contract/check layer** — IDS, bSDD, office standards, requirement packs.
2. **Inspectable data layer** — IFC/IfcOpenShell, Speckle objects/properties, graph representations.
3. **Adoption layer** — Rhino/Grasshopper tools that run checks without forcing full BIM behaviour.

The strategic people are those who bridge layers, not just those selling “AI for architecture”.

## Implications for Rhino Mind / Agents of Architecture

- Make Rhino Mind a **contextual checker/toolsmith**, not a chatbot.
- Treat IDS as inspiration for a small YAML/JSON check schema first; real IDS XML can come later.
- Use Speckle as the practical bridge: Rhino object metadata → Speckle properties → check report / project memory.
- Interview/follow implementers before product marketers. The best Top 100 people are the ones who have touched schemas, repos, connectors, and practice constraints.

### Mental Model Update

New models added to `people/mental-models.md` and `people/mental-model-notes.md`:

1. **Benchmark-first agentic BIM**
   - Before claiming a copilot works, define repeatable tasks, datasets, adapters, and evidence traces.
   - Rhino Mind implication: create a small benchmark for inspecting messy Rhino/Speckle models, extracting facts, running checks, patching metadata, and reporting provenance.

2. **One agent, many BIM adapters**
   - The LLM/agent should reason over stable task primitives while adapters translate into Rhino, Speckle, IFC, Revit, Bonsai, or web viewers.
   - Rhino Mind implication: define `rhino_mcp_adapter` primitives now, before building too many bespoke automations.

3. **Explore-then-check**
   - Because BIM/Rhino models are locally heterogeneous, agents should discover the project dialect before applying requirements.
   - Rhino Mind implication: first map layers, blocks, names, user text, Speckle properties, units, and missing metadata; then run IDS-lite checks.

4. **Semantics by doing**
   - Small teams will not maintain a separate semantic graph by hand. Semantics should be generated as a side effect of useful actions: asking questions, running checks, reusing details, and fixing model issues.

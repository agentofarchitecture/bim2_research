# Daily Note — 2026-05-23

## Executive read

BIM 2.0 is tightening around a practical stack: **openBIM contracts (IDS/bSDD/IFC) + model validation + project data platforms + domain LLMs**. The freshest signal is not “AI designs whole buildings”; it is that the industry is trying to make requirements, model semantics, and QA machine-checkable. For Rhino Mind / Agents of Architecture, the wedge is a BIM-light project intelligence layer: read Rhino/Grasshopper/Speckle/Revit-adjacent data, detect intent, and run small verifiable checks/tools.

### 4) Practice / business signals

- The product language is shifting away from “BIM database purity” toward **workflow outcomes**: QA, faster options, documentation automation, site analysis, design review, and data handoff.
- Small practice adoption still depends on: low setup cost, not forcing Revit-first behavior, clear audit trails, and outputs that can be checked by humans.
- Compliance/code-checking is attractive but dangerous as a first wedge. Better initial framing: “preflight”, “risk flags”, “missing information”, and “office standard checks”.

### 5) Adjacent tech

- GraphRAG / text-to-Cypher work supports the hypothesis that BIM 2.0 memory should expose entities, relationships, decisions, and evidence, not only chunks.
- Agentic tooling is useful only when bounded by a schema/checker. IDS is a promising target because it creates pass/fail criteria.
- USD/glTF/spatial data remain relevant as exchange/viewer layers, but the strategic moat is not file conversion; it is **semantic continuity between model, requirements, and decisions**.

## Pattern update

The structural shift today: **IDS is becoming the contract layer between human requirements and machine checking**. If IFC is the data exchange substrate, IDS is the test specification, bSDD is the controlled vocabulary, and project memory/RAG is the evidence layer. This is a better model for BIM 2.0 than “LLM directly edits BIM”.

## Implications for Rhino Mind / Agents of Architecture

- Rhino Mind should not try to become a full BIM authoring replacement. It should become a **semantic assistant around messy design work**.
- First-class actions should be small and verifiable: validate layers/blocks/names, detect missing data, generate checklists, map Rhino objects to lightweight classes, and export/round-trip through Speckle/IFC where useful.
- Treat IDS as a future output/input format: “generate an IDS-like requirement/check pack from a client brief or office standard”, then run it against model metadata.
- The UI should feel like contextual tools appearing in Rhino/Grasshopper, not a general chat box.

## Opportunity / experiment candidates

1. **IDS-lite checker for Rhino/Speckle**  
   Input: a small YAML/JSON requirement set. Output: pass/fail/warn report against Rhino object names/layers/user text/Speckle properties. Later map to real IDS XML.
2. **Brief/spec → model preflight checklist**  
   Use an LLM to extract requirements from a project brief, then produce human-readable checks and missing-information prompts. No legal compliance claims.
3. **Broken Rhino model triage**  
   Inspired by Arcol’s Rhino repair messaging: scan model hygiene, duplicated geometry, bad blocks, nonstandard layers, missing metadata, and produce fixes or Grasshopper/Python scripts.
